# Role Matrix and Access Layers Overview

This document provides a comprehensive overview of all roles within the Paroki Santo Klemens Digital Ecosystem, defining their `access_layer`, default landing pages per portal, and general responsibilities. It emphasizes how roles apply contextually within each portal, while core data (Family, Profile, Companion) remains accessible across portals based on underlying permissions.

## Core Principles of Role-Based Access:
-   **Gate Hub Redirection**: All Layer 2+ users enter via the Gate Hub upon login, then select their portal.
-   **Portal-Specific Experience**: Once a portal is selected, the user's role dictates their primary dashboard and available functionalities within that portal.
-   **Cross-Portal Data Access**: Access to general user data (e.g., family details, personal profile, Spiritual Companion) is maintained across all portals via global navigation elements (sidebar, navbar profile menu). Permissions for this data are determined by the user's underlying `access_layer` and specific RLS policies, not solely by the active portal context.
-   **Layered Access**: `access_layer` (0-10) defines the hierarchical level of trust and permissions, with higher layers having broader access.

## Access Layers Definitions:

| Layer | Description |
|-------|-------------|
| Layer 0 | **Publik**: Non-authenticated users. Access to public homepage and Bot 1 (public info bot). |
| Layer 1 | **Waiting Room**: Newly registered users pending KL approval. Restricted access to the Waiting Room page. |
| Layer 2 | **Umat Aktif**: Approved parishioners. Basic access to all portals (with specific functionalities), Family data, Profile, Digital Vault, Companion. |
| Layer 3 | **Wali Digital Lingkungan (WDL)**: Layer 2+ users delegated limited proxy access to assist other parishioners. |
| Layer 4 | **Ketua Lingkungan (KL)**: Manages a specific environment. Access to environment dashboard, pending user approvals, SOS restoration for their environment. |
| Layer 5 | **Sekretaris Paroki / Komsos / Seksos**: Administrative and social roles. Broader access to parish data, document verification, GAKIN data. |
| Layer 6 | **Bendahara Paroki / Koordinator Bidang**: Financial and program management. Access to financial dashboards, KPD/KTPD approvals, marketplace finance. |
| Layer 7 | **Koordinator Bidang**: Manages specific pastoral areas (e.g., Pewartaan, Liturgia). Oversees KPD/KTPD. |
| Layer 8 | **Wakil Ketua DPP**: High-level administrative oversight. Global GAKIN data, major approvals, audit oversight. |
| Layer 9 | **Pastor Paroki**: Highest pastoral authority. Global oversight, final approvals, Whistle-Blower access, SOS escalation. |
| Layer 10 | **Super Admin**: Developer/System administrator. Full system access (bypass RLS), monitoring, developer tools. |

-   **For Developers**: Understand `access_layer` mapping to features and data, guiding RLS policy implementation and frontend component rendering.
-   **For Product Owners**: Clarify role responsibilities and expected user experience per portal.
-   **For Testers**: Validate access control and ensure correct dashboard redirection and data visibility.

This matrix will be a living document, cross-referencing to more detailed role-specific documents within `docs/roles/` for deeper dives into responsibilities, permissions, and specific UI/UX details.
